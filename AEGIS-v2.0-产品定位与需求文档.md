# AEGIS v2.0 产品定位与需求文档

**版本:** 2.0  
**日期:** 2026-04-21  
**产品方向:** 多模型协作编程助手  
**状态:** 新定位基线文档

---

## 1. 产品定位

AEGIS v2.0 不再把自己定义为“严格门控工作流系统”。

新的产品定义是：

**AEGIS = 一个让 Claude、Codex 和其他模型自动分工协作的编程助手。**

它的核心价值不是：

- requirements lock
- gate review
- workflow state machine
- heavy governance

它的核心价值是：

- 让最合适的模型做最合适的事
- 把“写代码、审代码、拆任务、汇总结果”自动协作起来
- 在不增加用户负担的前提下提升编码速度和结果质量

一句话表述：

**当 Claude 写代码、Codex 补实现、多个模型并发处理子任务时，AEGIS 负责自动分工与整合。**

---

## 2. 我们不再做什么

以下能力不再是 v2.0 的主产品目标：

- 严格门控 workflow
- 强制 L1-L5 状态机
- 以 lock files 为核心的治理流程
- 以 Team Pack 为首页主入口
- 复杂的发布前审批链条

这些能力如果保留，也只能是：

- 兼容层
- 高级模式
- 后续可选插件能力

而不是当前产品主心智。

---

## 3. 我们真正要做什么

### 3.1 核心用户价值

用户最关心的不是系统内部怎么治理，而是：

1. 我输入一个编程任务后，系统能不能自动选模型
2. Claude 和 Codex 能不能配合得比单模型更好
3. 能不能在速度、质量、成本之间让我快速切换
4. 能不能少配置、少理解、直接用

### 3.2 核心产品承诺

AEGIS v2.0 要做到：

- 默认就是轻量协作
- 用户不需要理解底层编排
- 一个请求即可进入合适的协作模式
- 协作过程可见、可追踪、可复盘
- 在编程场景下明显优于单模型直跑

---

## 4. 目标用户

### 4.1 主要用户

- 日常使用 Claude Code / Codex 的开发者
- 想用多模型提升编码效率的个人开发者
- 需要代码生成 + 代码审查 + 并发拆解能力的小团队

### 4.2 典型用户场景

- “帮我实现这个功能”
- “先让一个模型写，再让另一个模型审”
- “把这件事拆成几个子任务并发做”
- “给我多个方案，然后汇总成最好的答案”
- “修 bug 时让一个模型定位、一个模型修、一个模型复查”

---

## 5. 产品主能力

### 5.1 自动模型分工

系统根据任务类型自动选择：

- Claude 做复杂推理、架构、复审
- Codex 做代码生成、快速实现、修复
- 其他模型做补充、低成本任务或本地隐私任务

### 5.2 协作模式

AEGIS v2.0 的主能力围绕 4 个模式展开：

#### 1. Pair

一个模型写，另一个模型审。

适合：

- 功能开发
- 重构
- 风险较高的代码修改

#### 2. Swarm

把任务拆开，多模型并行完成，再自动聚合。

适合：

- 生成测试
- 并行分析多个文件
- 大任务拆解

#### 3. Pipeline

按阶段顺序执行。

适合：

- 分析 → 修复 → 验证
- 设计 → 编码 → 评审

#### 4. MoA

多个模型独立给答案，再由聚合模型整合。

适合：

- 方案评审
- 架构比较
- 高不确定性问题

### 5.3 轻量会话管理

系统需要保留：

- session
- stage outputs
- shared context
- messages
- checkpoints

但这些是“支撑协作”的基础设施，不是产品首页卖点。

### 5.4 成本 / 速度 / 质量模式

用户应能明确切换：

- `speed`
- `balanced`
- `quality`
- `cost`

这是产品核心体验的一部分，不是隐藏配置。

---

## 6. 产品边界

### 6.1 v2.0 内必须完成

- Model Registry
- Task Router
- Pair / Swarm / Pipeline / MoA
- Session / Message / Context 基础设施
- CLI 主入口
- 基础成本可见性
- 多模型 bridge 能力

### 6.2 v2.0 不强求完成

- 复杂治理状态机
- 严格 lock file 体系
- 重型 Team Pack 产品面
- Evolution / Nightly Schedule
- 企业级流程审批

### 6.3 可作为后续高级能力

- 高级 governance
- 可插拔团队模式
- IDE 插件
- 学习优化路由
- 企业审计与合规能力

---

## 7. 产品体验原则

### 7.1 默认简单

默认用法应该尽可能像这样：

```bash
aegis "实现 JWT 登录"
aegis pair "重构认证模块"
aegis swarm "生成测试用例"
```

而不是让用户先理解 workflow、gate、registry lock。

### 7.2 自动优先

系统应该自动决定：

- 用哪个模式
- 用哪些模型
- 是否并发
- 如何汇总

用户只在需要时覆盖。

### 7.3 可解释但不啰嗦

用户应该能看到：

- 为什么选这个模式
- 为什么选这些模型
- 每个模型做了什么

但不应该被内部实现细节淹没。

### 7.4 编码场景第一

所有主决策都以编程场景为中心：

- 写代码
- 改代码
- 审代码
- 修 bug
- 生成测试
- 评估方案

## 9. CLI 产品形态

建议把 v2 CLI 收敛到下面这套心智：

```bash
aegis "实现登录功能"
aegis run "修复 SQL 注入"
aegis pair "重构认证模块"
aegis swarm "生成测试用例"
aegis moa "评审认证方案"
aegis models list
aegis session list
aegis cost report
```

说明：

- `run` 是默认智能路由入口
- `pair/swarm/pipeline/moa` 是显式协作入口
- `models/session/cost` 是辅助查看入口
- 不让用户优先接触旧 workflow 命令

---

## 10. 成功标准

AEGIS v2.0 的成功标准不再是“流程治理是否严密”，而是：

### 10.1 产品成功标准

- 用户能在 1 分钟内理解产品是做什么的
- 默认命令能直接解决编程任务
- 多模型协作体验明显优于单模型直跑

### 10.2 功能成功标准

- 路由可信
- 协作模式可执行
- 显式模型选择真实生效
- session 可追踪
- 成本基础可见

### 10.3 体验成功标准

- 命令简洁
- 错误可理解
- 文档与实现一致
- 不让用户先学内部治理模型

---

## 11. 与旧方向的关系

旧方向不是完全废弃，而是降级处理：

| 旧能力 | 新定位 |
|------|------|
| Workflow | 兼容层或高级模式 |
| Governance | 可选能力，不做主叙事 |
| Team Pack | 后续扩展能力 |
| Lock Files | 内部机制，不对外主打 |
| Heavy Review Loop | 收敛为 Pair / Pipeline 的一种内部实现 |

换句话说：

**AEGIS 先成为好用的多模型编程助手，再考虑是否保留复杂治理能力。**

---

## 12. 当前阶段的产品结论

从现在开始，AEGIS v2.0 的重构目标应统一为：

**把现有项目从“工作流控制系统”重构成“多模型协作编程助手”。**

所有后续技术和产品决策，都应优先回答这个问题：

**它是否直接提升了多模型协作编程的效果、速度或易用性？**

如果没有，就不应继续占据 v2.0 主路线。
