# AEGIS OS 市场研究报告

## 研究目标

分析 Git-Native 多 Agent 协作工作流系统的市场机会，为 AEGIS OS 的产品定位与后续 PRD 提供决策依据。

---

## 1. 市场背景

### 1.1 多 Agent 系统正从"Demo 阶段"进入"治理阶段"

2024-2025 年是 Agent 框架爆发期，AutoGen、CrewAI、LangGraph 等项目快速积累了开发者关注。但行业共识正在转移：

- **第一阶段问题已解决**：如何让多个 Agent 对话、分工、调用工具。
- **第二阶段问题凸显**：Agent 在复杂、长期、多步骤执行中如何"不跑偏"。

这一转移创造了"Agent 治理层"的新市场空间。

### 1.2 "个人公司"（Company of One）成为新兴用户画像

随着 AI 编码、研究、写作、运营能力的普及，越来越多的独立开发者、超轻创业团队开始把 AI 当作"虚拟团队成员"。这类用户的核心痛点不是"缺能力"，而是：

- AI 做着做着改了自己的需求理解
- 多轮执行后偏离原始目标
- Review 只是建议，不是约束
- 没有可追溯的决策和审计链

AEGIS 将目标用户明确定位为"个人公司"，与当前大多数面向企业 IT 部门的 Agent 平台形成差异化。

---

## 2. 市场规模与趋势

### 2.1 相关市场规模（估算）

| 领域 | 市场规模 | 说明 |
|------|----------|------|
| AI Agent 平台/框架 | ~$15-25B（2025 估算） | 涵盖低代码 Agent 构建、企业 RPA+AI、开发者框架 |
| AI 辅助软件开发 | ~$8-12B | GitHub Copilot、Cursor、Codex 等 |
| 个人开发者/Solopreneur 工具市场 | ~$3-5B | 独立建站、无代码、自动化、内容生产工具 |

AEGIS 所在的是"Agent 治理/工作流 orchestration"细分市场，目前尚无统一定义，但处于高速增长期。

### 2.2 关键趋势

1. **从 Chat 到 Workflow**：用户不再满足于单次问答，而是需要把复杂目标交给 AI 持续执行。
2. **Git-Native 成为工程化 Agent 的默认选择**：把 Agent 执行过程版本化、可审计，逐渐被认可为最佳实践。
3. **Host-Native 模式崛起**：Agent 直接运行在 Claude Code / Codex / Cursor 等宿主环境里，而不是外部 CLI 再递归调用模型。
4. **需求漂移（Drift）成为核心焦虑**：社区讨论重心从"Agent 能做什么"转向"Agent 会不会自己做主改需求"。

## 3. 用户细分

### 3.1 核心用户：个人公司 / 超轻团队

**特征**：
- 1-3 人团队，或独立开发者
- 使用 Claude Code / Codex / Cursor 作为日常开发环境
- 有多个并行的产品/业务想法
- 时间有限，无法每一步都盯着 AI
- 对"失控"高度敏感

**需求优先级**：
1. 目标锁定（AI 不要自己改需求）
2. 自动推进（减少人工干预）
3. 可审计（能回溯决策过程）
4. 可扩展（未来能接更多能力）

### 3.2 次要用户：小型技术团队

**特征**：
- 5-15 人工程团队
- 希望引入 AI 辅助代码审查、PRD 撰写、安全审计
- 需要规范化的 workflow 而不是 ad-hoc 的 prompt 工程

**需求优先级**：
1. 角色分工明确（Research -> Plan -> Build -> Review -> Deploy）
2. Gate 机制有约束力
3. 与现有 Git 流程兼容
4. 可自定义 agent 和 review 规则

---

## 4. 关键机会与风险

### 4.1 机会

- **空白市场**：专注于"个人公司"的 Agent 治理层产品极少，大多数竞品面向企业或纯开发者。
- **先发优势**：Git-Native + Host-Native 的组合定位在市场中具有独特性。
- **生态红利**：Claude Code / Codex 等宿主能力的增强，直接利好 host-native 的 AEGIS 架构。
- **长期粘性**：一旦用户建立起 Owner Profile 和 Company Profile，迁移成本高。

### 4.2 风险

- **认知教育成本**："Agent 治理"概念较新，用户可能先理解为"又一个 Agent 框架"。
- **竞品快速跟进**：AutoGen、CrewAI 等也在探索 workflow 和 memory 能力。
- **宿主依赖**：Claude Code / Codex 的 skill/API 变化可能影响 host-native 接入方式。
- **使用门槛**：AEGIS 的规范化流程对轻量用户可能显得"重"。

---

## 5. 结论与建议

1. **AEGIS 选择"个人公司 + 治理优先"的差异化定位是正确的**，与现有竞品形成错位竞争。
2. **Host-Native + Git-Native 是核心护城河**，应持续强化这一架构叙事。
3. **早期用户应从已有 Codex/Claude Code 使用习惯的开发者切入**，他们对 workflow 和 review 的价值感知最强。
4. **需警惕"过度工程化"感知**：在保持治理约束的同时，应尽可能降低首次使用门槛。
